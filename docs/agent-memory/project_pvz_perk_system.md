**存档崩溃修复（2026-06-14，commit `3d3b5fd`，直接 master）**：零词条生存档（survivalRound 1 / CHOOSE_CARD 未选词条）读档必崩。**两端 nlohmann 双 foot-gun**：①写端 `Save(j["perks"])`——nlohmann 可变 `operator[]` 仅被求值就把 `"perks"` 键**物化成 null**，而 `Save` 只写 `stacks>0` 的键，零词条时啥都不写→磁盘留 `"perks": null`。②读端 `contains("perks")` 守卫对 null **无效**（键存在只是值为 null），`Load` 里 `j.value(key,0)` 在 null 上**抛 `type_error.306 "cannot use value() with null"`**→CrashHandler。修法（defense in depth，两端各治本）：写端改先 Save 到局部 json、`!is_null()` 才 `j["perks"]=...`（零词条省略键，等同旧档）；读端 `Load` 首行 `if(!j.is_object()) return;`（兜住已落盘的 null 旧档，Load 是契约属主）。**通用教训**：本仓库凡 `Save(j["key"])` 范式（perk 存档契约就是这个）都有同款 null 物化风险，`contains()` 不能挡 null，`value()`/`at()` 在 null 上会抛——加词条/存档时照此双端守。验证：perk manager 自包含（仅 `<array>`+json），clang++ 单独编 `SurvivalPerkManager.cpp`+一次性 test 跑出 RED→GREEN，并直接读真实 `level1000_data.json`(perks=null) 复跑读路径确认不抛。

**存档异常安全边界（同会话续作，commit `c547409`，master）**：主人要求给 `GameInfoSaver` 存/读档统一加 try/catch 兜底，避免以后任何损坏存档直接崩游戏。做法 = 四个公有接口（Save/LoadPlayerInfo、Save/LoadLevelData）改薄包装，实际逻辑下沉到对应 `*Impl`，公有层只 `try{return XImpl();}catch(const std::exception& e){LOG_ERROR;return false;}`（沿用既有"失败=false"契约，不逸出到 CrashHandler）。**关键 gotcha（踩过）**：`*Impl` 必须是 `GameInfoSaver` 的**私有成员**，不能是 .cpp 自由函数——`Board.h:68` 是 `friend GameInfoSaver;`（只 friend 本类），`mIsLoadSave` 等是 private，自由函数编译期报 `private member of 'Board'`。`*Impl` 后缀避免与同名公有方法无限递归（成员函数内无限定名调用会解析到自身）。catch 选 `std::exception`（nlohmann 全异常树 type_error/parse_error/缺键 out_of_range 都派生于此）而非 `catch(...)`（后者会吞访问违例等真致命）。**遗留**：`LoadLevelData` 中途抛异常时 Board 已部分加载不回滚，false 仅保证不崩、调用方须按全新关卡处理（已写进代码注释）。AutoTest 短路存档→catch 路径无法运行时触发，靠异常树继承关系+编译+smoke 回归保证。

GetPlantAttackSpeedMultiplier() : 1.0f`。**核心非显然教训（读 PerkType.h 看不出，必记）= 间隔/动画耦合**：开火计时器在射击动画**开始**时清零，子弹经射击动画的 frame event（Shooter 第 64 帧 / PuffShroom 第 28 帧）吐出。所以**必须两处一起缩放**：间隔阈值 `mShootTime / mult`（更频繁）**且** 射击动画 clip-speed `字面量 × mult`（吐弹帧等比前移）。只砍间隔不加快动画→当 `有效间隔 < 动画跑到吐弹帧的时间`，动画在吐子弹前被反复重启→**一颗子弹打不出**（高层数下的真 bug）。Repeater 补发分支(`mPendingSecondShot`,`2.0f×mult`)也要乘——易漏，code-review 抓到补注释。`mBoard` 空守卫防预览植物（AlmanacScene 用 board=nullptr 建预览）。存档零改动（倍率从层数实时派生，`mShootTimer` 已随植物 SaveExtraData 落盘）。AutoTest：`shootIntervalOn1500=round(1500/mult)` 整数化闭合断言（2层→1154，规避浮点等值），`smoke_perks_attackspeed.json`。配对池：现 3 植物增益(+攻速) vs 4 僵尸增难，`RollPerkPairings` 笛卡尔积自动容纳非对称，无需新增僵尸诅咒。走 brainstorm→spec→plan→subagent-driven 全流程（2 Task 各两阶段审查+终审），feature 分支合后删。spec/plan `docs/superpowers/{specs,plans}/2026-06-22-survival-perk-plant-attack-speed*`。**捎带**：同会话先把主人未提交的 PLANT_REGEN 平衡改动**单独提交**（`2f949c7`：25HP/5层→**65HP/7层**，过量治疗解锁改回 `>=5`）；后续又把 maxStacks 调到 8，故当前代码为 65HP/层、最多8层、5层解锁过量治疗。
