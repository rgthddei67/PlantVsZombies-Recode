**Foot-guns（本次实抓，将来同型改动直接查）**：
1. **onTriggerStay 绕过 lambda 守卫**（终审才抓）：Polevaulter 只重写 onTriggerEnter 的 lambda 加 RUNNING 守卫，但基类 onTriggerStay 仍调虚函数 StartEat → 跑动撑杆停下啃魅惑。修=override `Polevaulter::StartEat` 在虚函数层拦（虚函数覆盖所有 trigger 路径，lambda 只覆盖一条）。**教训：给碰撞回调加行为守卫，放虚函数不放 lambda。**
2. **Edit old_string 撞形近行**：大喷菇豁免被 Edit 加进 HasZombieInRow（触发检测）而非 FumeAttack（扣血）——两函数判定行文本几乎相同；且单僵尸测试场景喷雾根本不开火 → 假阴性通过。修后脚本必须双僵尸（魅惑 270 未损+普通掉血=实证开火）。**教训：同文件有形近行时 Edit 要带更多上下文；豁免类测试必须先证明"攻击确实发生了"。**
3. **JUMPING 撑杆理论可啃魅惑**（终审备忘，未修）：守卫只排 RUNNING；窗口极窄，要修一行改 `mVaultState != WALKING`。
4. **纸僵尸(及继承的 FastPaperZombie)被魅惑后卡死 anim_eat**（后续修复 d83bab0，2026-07-02 已合 master 未 push；主人问 WalkTrackAfterEat 时揪出）：基类魅惑/啃食收尾（`StartMindControlled`/`EatTarget`/`StopEat` 啃僵尸分支/`ValidateEatingState`）硬编码回落 `WalkTrackAfterEat()="anim_walk2"`，但 PaperZombie.reanim 只有 `anim_walk`/`anim_walk_nopaper`→`Animator::PlayTrack` 遇缺失轨道**静默 return false、不改动画状态**→魅惑 `mindControlled=true` 却卡在 `anim_eat` 不走路(位置也不动)。纸僵尸只覆写了自己的 StartEat/StopEat/ValidateEatingState（植物路径），**啃僵尸分支委托回基类**故这些路径漏掉；撑杆靠覆写 `WalkTrackAfterEat="anim_walk"` 躲过、纸僵尸躲不过。**修=基类新增 `virtual void ResumeWalkAfterEat(float blendTime)`（默认实现 `PlayTrack(WalkTrackAfterEat(),0.0f,blend)` 对有 anim_walk2 的僵尸逐位等价、零回归），4 处收尾改走它；PaperZombie 覆写它按 `mHasNewspaper` 选 anim_walk/anim_walk_nopaper 并带 `kNoPaperWalkClip`。** 教训：①「子类只覆写部分虚函数、另一分支委托回基类」时，基类里的硬编码假设（轨道存在/anim_walk2）会**只对该子类**失效——查覆盖面要连委托路径一起看；②不能图省事只覆写 `WalkTrackAfterEat()`——`const char*` 带不了 clip 速度，狂暴纸僵尸被魅惑后会走异常慢，故抽 `ResumeWalkAfterEat` 才是修根因。复现脚本 `smoke_charm_paper_eating.json`（纸僵尸啃坚果时 charm；dump_state 的 `track` 字段作断言：修前卡 anim_eat/修后 anim_walk），回归 smoke_charm_{basic,bite,polevaulter,fume} 全绿、clang-release 0 warning。
5. **铁门僵尸被魅惑后多出一条手臂**（后续修复 c0cb799，2026-07-02 已合 master 未 push；主人截图报的 bug，与 #4 同源）：DoorZombie 啃植物时 `StartEat` 会 `ShowArm(true)` 露出常规手臂来啃（平时藏门后），正常停食由 `DoorZombie::StopEat`/`ValidateEatingState` 调 `ShowArm(false)` 藏回；但被魅惑走基类 `StartMindControlled → ResumeWalkAfterEat`，**该路径绕过 DoorZombie::StopEat**，只清 mIsEating+播走路、从不藏手臂 → 常规臂 + 持门臂并存 = 多一条。**修=DoorZombie 覆写 `ResumeWalkAfterEat`：先调基类(播 anim_walk2)，再在 `mShieldType != NONE`(门还在)时 `ShowArm(false)`——与自身 StopEat 逐字同构；门已掉(ShieldDrop 把 mShieldType 设 NONE)则跳过、保持手臂可见。** 非魅惑的啃植物收尾仍走 StopEat/ValidateEatingState(未改)，零回归。**可推广规律：任何"啃食期改视觉状态(露臂/换贴图)"的僵尸子类，都必须覆写 `ResumeWalkAfterEat` 做对称清理，否则魅惑/啃僵尸收尾路径会漏——`ResumeWalkAfterEat`(#4 引入) 现是这类清理的统一挂载点。** 验证抓手：`TestDriver` dump_state 新增 `armVisible` 字段（`GetTrackVisible("Zombie_outerarm_hand")`，无此轨道的僵尸安全返回 false，是手臂显隐类 bug 的通用断言）；`smoke_charm_door_eating.json`（修前 charm 后 armVisible 恒 true+截图见多余臂/修后 false+目视手臂消失）；回归含 paper_eating 共 5 脚本全绿、0 warning。**存档侧核查过无独立 bug**：mIsEating 持久化（Zombie.cpp:107/126）+ ZombieItemUpdate 的 `else if(shieldGone||mIsEating)` 恢复手臂，且 AutoTest 短路存档无法直测（靠 inspection）。
6. **铁门魅惑后啃敌方僵尸时无手臂**（52bd86d，紧接 #5，同一"门臂 show/hide 须成对"故事）：铁门手臂要 `StartEat` 露臂 + 收尾藏臂**成对**。植物分支两半齐全（StartEat 的 ShowArm(true) + StopEat/ValidateEatingState 的 ShowArm(false)）；但**僵尸分支（`DoorZombie::StartEat` 中 `OBJECT_ZOMBIE` 那条，仅"魅惑后啃敌方僵尸"走）只 `Zombie::StartEat(other); return;`、从不 ShowArm(true)**。#5 之前"多一条手臂"的 bug 歪打正着让魅惑铁门一直显臂；#5 把收尾藏臂修对后，show 这半的缺失就暴露成"魅惑啃食时手臂没了"（主人 NEW BUG2 现象，说"现在"正因此）。**修=DoorZombie::StartEat 僵尸分支补 `ShowArm(true)`（守卫 `!wasEating && mIsEating && mShieldType!=NONE`，与植物分支 / ResumeWalkAfterEat 三处对称；门已掉则手臂恒显、跳过）。** 非魅惑不啃僵尸故零影响。**教训升级：排查子类"啃食期视觉状态"时，show 和 hide 两半、且"植物分支/僵尸分支(魅惑专用)"多条入口都要各自补齐——只补一半会被另一个 bug 掩盖，直到那个 bug 修好才现形（本例正是被 #5 未修前的多手臂 bug 掩盖）。** 验证：`smoke_charm_door_eat_zombie.json`（魅惑铁门啃普通僵尸，anim_eat 时 armVisible 修前 false/修后 true+截图目视手臂伸出）；回归 door_eating(啃植物→魅惑) + charm_{basic,bite,polevaulter,fume,paper_eating} 全绿、0 warning。
