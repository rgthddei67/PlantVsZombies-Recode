# 冒险出怪表编排

用于新增僵尸进入冒险模式，或重排一整大关的 `spawnlists.json`。目标是让玩家逐步认识威胁，同时让正式出波抽取与选卡预览都保持清晰。

## 先收集完整约束

1. 从 `AdventureProgression.h` 重新确认 `LEVELS_PER_AREA`、大关边界和每关通关奖励。内部绝对关卡号必须由当前代码换算，不能照抄旧记忆。
2. 把目标大关每关的 `waves`、`sun`、僵尸 ID 数组和类型名列成表；同时列出每种僵尸的 `weight`、`appearWave`、是否为召唤/变异单位。
3. 区分“通关本关后获得”的植物与“进入本关时已经拥有”的植物。安排克制关系时只按后者计算。
4. 标记重点威胁：新机制、快速突进、高护甲、召唤、天气变异、破坏小推车等；普通/路障等基础怪不算独立教学主题。
5. 确认只编辑 `build/clang-release/resources/spawnlists.json` 权威文件；其他 preset 必须是通过 Junction 看到同一内容。

## 编排规则

- 每关最多首次教学一个重点威胁。同家族强化变体必须在基础型至少提前一关出现后再引入，不能同关首次亮相。
- 重点威胁的首次关使用“基础杂兵 + 少量旧机制 + 本关主题”的精简池；撤下相邻重点怪，让玩家能辨认失败原因。
- 两个重点威胁首次关之间可以相邻，但首次关不得同场；综合前至少安排一关让前一个威胁暂退，形成一次无同场复习。
- 最终关组合已经学过的威胁，但不要机械收录全部已实现类型。过大的池会稀释重点怪的实际抽中率，也会挤乱选卡阶段的预览队列。
- `appearWave` 必须小于等于关卡波数，并为新机制留下足够实际出场波次；短关里把 `appearWave: 8` 的怪放进池，不等于玩家能充分学会它。
- 波数应随大关总体平滑递进。非 10 倍数会改变旗帜数量与最终波位置，只有明确需要时才使用并通过 UI 截图确认。
- 召唤单位、天气变异体等 `weight: 0` 类型不得写入出怪池；只放能被正常扣点生成的基础类型，由其既有 resolver/召唤逻辑派生。
- JSON 数组顺序同时影响预览顺序，保持“基础怪 → 旧机制 → 本关重点”稳定排列，避免无意义改序制造 diff。

## 静态校验

修改后逐项确认：

1. 权威 `spawnlists.json` 可解析，目标关卡各出现一次，`zombies` 非空且内部无重复。
2. 每个 ID 都落在 `[0, NUM_ZOMBIE_TYPES)`，并能映射到已注册工厂；池中不存在 `weight <= 0` 类型。
3. `clang-playtest`/`msvc-debug` 的 `resources` 路径必须是指向 clang-release 权威目录的 Junction，不再比较或提交副本。权威 JSON 位于默认忽略目录，提交时使用 `git add -f` 并检查 staged diff。
4. 输出一张“关卡｜波数｜首次主题｜完整池｜进入时可用关键植物”的最终表，人工检查首次登场与综合间距。

## AutoTest 与可见验收

- `dump_state`/`assert_state` 至少覆盖 `level`、`background`、`maxWave`、`sun`、`spawnTypeCount` 和完整 `spawnList` 数组；按整大关逐关切换，不只测改动最大的末关。
- 对每个重点威胁首次关与最终综合关截图，检查选卡预览有没有重叠、越界或主题不清。
- 按 `PROJECT_GUIDE.md` 从 `build/<preset>/` 在主人当前桌面可见运行，保留 exit code、`run.log`、逐关状态 JSON 和截图。
- 验证通过后，把最终关卡表和设计理由更新到项目记忆；若旧主题文件声称“全阵容”或记录了旧首次登场关，同步改掉过期结论。
